**Стандартная монада State**

Хотя всем мы любим функции за то, что они чисты, предсказуемы и не зависят от внешнего состояния, часто это самое состояние необходимо должно влиять на работу функций и наоборот - функции должны менять это состояние. Это всё делается с помощью монады `State`, не нарушая чистоту функций - она автоматически протаскивает состояние через цепочку вычислений, скрывая явную передачу аргументов.

```haskell
class Monad m => MonadState s m | m -> s where
  get :: m s
  put :: s -> m ()
  state :: (s -> (a, s)) -> m a

newtype State s a = State { runState :: s –> (a, s) }
```
, где `s` - тип состояния, 
`a` - результат вычислений с состоянием.

Функция `get` возвращает текущее состояние,а ункция `put` меняет текущее состояние:
```haskell
get = state $ \s –> (s, s) -- возвращает состояние и как результат тоже
put newState = state $ \s –> ((), newState) -- возвращает новое состояние без результата
```
Функция `state` эквивалентна конструктору `State` - она берёт вычисление с состоянием и оборачивает его в монаду.
Так же полезными функциями являются `evalState`, которая возвращает только результат вычисления, и  `execState`, которая возвращает только состояние, и `modify`, которая применяет функцию к состоянию и меняет его:
```haskell
evalState :: State s a -> s -> a
execState :: State s a -> s -> s
modify :: MonadState s m => (s -> s) -> m ()
```

Пример:
```haskell
import Control.Monad.State.Strict

increment :: State Int ()
increment = modify (+1)

counter :: State Int Int
counter = do
  increment
  increment
  increment
  get  -- вернуть текущее значение

-- Запуск:
-- evalState counter 0  => 3
-- execState counter 10 => 13
-- runState  counter 5  => (8, 8)
```

Есть две версии `State` - ленивая (в `Control.Monad.State.Lazy`) и строгая (`Control.Monad.State.Strict`). Ленивая версия лениво меняет состояние, из-за этого может быть утечка на накопление `thunk`.
